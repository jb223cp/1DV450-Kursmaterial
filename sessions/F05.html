Vad är REST?

  • Varje "sak" i vår tjänst/applikation är en resurs (resource)
  • Varje resurs måste kunna adresseras (URI/addressability)
  • En representation beskriver en resurs tillstånd (state)
  • En resurs kan ha flera representationer (mutiple representations)
  • REST är tillståndslöst på servern (stateless)
  • Hypermedia as the engine of application state (HATEOAS)
  • Bygger på de arkitektoniska principerna för webben

POST vs. PUT

  • POST är vagt definierad i HTTP-specifikationen, "Data-handling process"
  • POST-to-Append - POST används för att skapa nya resurser
  • Overloaded POST - POST används till allt utom get (HTML-formulär)
  • PUT - Modifiera en resurs tillstånd. Kan skapa om man skickar med nytt id. Vill man det?

Idempotent vs. Safe Methods

Metoder som kan anropas flera gånger efter varandra och ge samma resultat kallas idempotenta.
PUT /products/123 {"id": 123, "product_name": "Trolldeg"}

Vad händer om vi får en timeout? Har resursen ändrats?
Responsen kan vara annorlunda mellan olika anrop
Första gången: 204 (No Content)
Resterande gånger: 404 (Not found)

GET, HEAD anses som safe methods då dessa inte förändrar en resurs

GET /products/123/delete
DELETE /products/123

Internet är ett osäkert nätverk. Timeouts.
Du vet inte om din DELETE slog igenom.

Vilka metoder är idempotenta?

  • GET
  • POST
  • DELETE
  • PATCH

PUT skickar alltid en ny represenation = indempotent PATCH kan appliceras på flera sätt och kan vara idempotent men man vill hålla den fri från det kravet PATCH {change: 'Name' from: 'benjamin franklin' to: 'john doe'}

Multipla representationer

  • En representation av en resurs kan ges i flera dataformat
  • Klienten ska kunna efterfråga det dataformat man önskar via sin HTTP förfrågan
  • Anges inte detta av klienten bör resursen returneras i ett "default" dataformat
  • I sitt svar talar servern om i vilket dataformat svaret är (Content-Type)

HATEOAS

Hypermedia As The Engine Of Application State

The model of application is an engine that moves from one state to another by picking alternative state transitions in current set of representations

  • Current state - Beskrivs av representationen och dess dataformat
  • Länkar/URL:er - Beskriver "the transition" till nästa "state"
  • Inkludera länkar/URL:er i ditt APIs response
  • Klienter bör kunna navigera sig i ditt API precis som man surfar
  • En klient ska aldrig behöva bygga ihop en URL
  • "Hypertext should be used to find your way through the API."

Foto from: Foto cc by: https://blog.apigee.com/detail/api_design_honing_in_on_hateoas

Varför HATEOAS?

  • Bygger på samma koncept som webbens grundpelare - URL, HTTP, HTML (hyperlänkar)
  • Gör det enklare för klienten - Ska inte behöva bygga ihop URLer, kan skriva mer automatiserad kod
  • Flexibelt för förändringar, förändringar kommer ske!
  • Förväntade routes kommer brytas!
  • HATEOAS reducerar detta, lösare koppling mellan server och klient
